Build Prompt: Majlis Group Chat & Direct Messaging System
What to Build
Add full group chat and peer-to-peer (DM) messaging to the existing Majlis private forum at /majlis. Currently the Majlis has a single shared chat room with mock data on the frontend. Replace this with a real, multi-channel system: users can create/join groups and DM other members directly. All messages must be encrypted before storage in the existing PostgreSQL database (Drizzle ORM — NOT Supabase).

Center: Active chat view (the existing message UI — keep the MessageBubble component style, reply threading, scroll-to-bottom button).
Right sidebar: Members panel (already exists) — add a "Message" button next to each member to start a DM. Add a "Create Group" button at the top.

Update majlis_messages: Add a channelId (integer, FK → majlis_channels.id, nullable for backward compat). Messages without channelId are legacy "General" room messages. All new messages must have channelId.

Message Encryption
Encrypt messages before storing in DB. Approach:

Use AES-256-CBC encryption with a server-side encryption key stored as an environment variable (MAJLIS_ENCRYPTION_KEY — 32-byte hex string).
Encrypt message content server-side before INSERT. Store the encrypted ciphertext + IV (prepend IV to ciphertext, both hex-encoded, separated by :) in the content column.
Decrypt server-side when reading messages before sending to client.
This is symmetric encryption (not end-to-end) — the server can read messages but the database at rest contains only ciphertext.
Use Node.js built-in crypto module: crypto.createCipheriv('aes-256-cbc', key, iv) / crypto.createDecipheriv(...).
Generate a random IV per message for security.
Do NOT use salted hashing (per user request) — use deterministic symmetric encryption with per-message IV.
API Endpoints to Add (in artifacts/api-server/src/routes/majlis.ts)
All require requireMajlisAuth middleware:

Channels:

POST /majlis/channels — create a group channel (body: { name, description?, memberIds[] }). Creator auto-added. For DM: { type: "dm", memberIds: [otherUserId] } — enforce exactly 2 members, check no duplicate DM channel exists.
GET /majlis/channels — list channels the current user belongs to. Return last message preview (decrypted), unread count, member count.
GET /majlis/channels/:id — get channel details + members list.
POST /majlis/channels/:id/members — add members to a group (body: { userIds[] }). Only group creator can add. Not allowed for DM channels.
DELETE /majlis/channels/:id/members/:userId — leave a channel (users can leave groups; can't leave DMs, only archive).
Messages (update existing):

GET /majlis/channels/:channelId/messages — paginated messages for a channel (replaces current /majlis/messages). Decrypt before returning.
POST /majlis/channels/:channelId/messages — send message to a channel. Encrypt before storing.
GET /majlis/channels/:channelId/messages/poll?after=ID — poll for new messages in a channel.
Keep legacy /majlis/messages endpoint working for backward compat (treat as "General" channel).

Edge Cases to Handle
DM deduplication: if user A messages user B, and user B later tries to message user A, open the existing DM channel instead of creating a new one.
Banned/muted users: banned users can't access any channel. Muted users can read but not send.
Empty channels: show a welcome/empty state.
Channel with deleted messages: handle isDeleted flag in message display.
Legacy messages (no channelId): migrate or display in "General" channel.
Encryption key missing: fail loudly on server startup if MAJLIS_ENCRYPTION_KEY env var is not set. Provide a script/instruction to generate one (node -e "console.log(require('crypto').randomBytes(32).toString('hex'))").
